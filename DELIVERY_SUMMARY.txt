KEY FINDINGS
================================================================================

ARCHITECTURAL PATTERNS:
- Service-agnostic container (IFluidContainer)
- Service-specific clients (AzureClient, TinyliciousClient)
- Strongly typed schemas using generics
- Event-driven state management
- Member aggregation by userId (not clientId)

CRITICAL METHODS:
✓ createContainer<T>(schema: T, compatibilityMode: "1" | "2")
✓ getContainer<T>(id: string, schema: T, compatibilityMode: "1" | "2")
✓ attach(): Promise<string> (returns container ID)
✓ connect() / disconnect()
✓ create<T>(objectClass: SharedObjectKind<T>)

CRITICAL EVENTS:
✓ connected, disconnected, saved, dirty, disposed

CRITICAL INTERFACES:
✓ IFluidContainer<T extends ContainerSchema>
✓ IServiceAudience<M extends IMember>
✓ ITokenProvider (with fetchOrdererToken/fetchStorageToken)

TYPE SAFETY:
✓ Generic type parameters preserve container.initialObjects types
✓ ContainerSchema defines both initialObjects and dynamicObjectTypes
✓ Service-specific member types extend base IMember

AUDIENCE SEMANTICS:
✓ getMembers() returns Map<userId, member>
✓ Single user can have multiple connections
✓ Only interactive connections included by default
✓ getMyself() returns current user + currentConnection

VERSION MANAGEMENT (Azure only):
✓ viewContainerVersion() returns read-only container
✓ getContainerVersions() returns list with maxCount
✓ Default max: 5 versions

================================================================================
COMPARISON HIGHLIGHTS
================================================================================

AzureClient vs TinyliciousClient:

AZURECLIENT:
- Required constructor props (connection config required)
- Supports remote AND local connections
- Version management (viewContainerVersion, getContainerVersions)
- Custom logger support
- Custom config provider support
- Storage compression options
- Member names provided by Azure

TINYLICIOUS:
- Optional constructor props (has defaults)
- Local development only (Tinylicious service)
- No version management
- No logger/config customization exposed
- Simpler API
- Member names generated locally

BOTH:
- Same createContainer/getContainer signatures
- Same IFluidContainer interface
- Same audience member aggregation
- Same ContainerSchema support
- Same CompatibilityMode support
- Same token provider interface
